## 第 2 轮审查: CEO Review (HOLD SCOPE)

> 评审模式: HOLD SCOPE — 不增减功能，专注于让现有计划无懈可击
> 审查范围: 架构深度审查、边缘情况覆盖、文档一致性、测试覆盖缺口

### 发现 1: useTimer 返回类型不一致 🔴

**问题**: PRD 声明返回 `{ state, start, pause, reset }`（state 对象包装），但架构文档组件树中 App 的解构是扁平的 `{ timeLeft, mode, totalTime, isRunning, start, pause, reset, progress }`。两套 API 签名无法同时成立。

**影响**: 实现者无法确定调用方式，会导致组件 props 传递错误。

**修复**: 统一为扁平返回结构，符合 React Hooks 惯例（useState 也是扁平返回）。useTimer 返回类型改为：

```typescript
{ timeLeft, mode, totalTime, isRunning, progress, start, pause, reset }
```

### 发现 2: 计时策略矛盾 🔴

**问题**: 架构文档承诺使用基于 `Date.now()` 的时间差计算（应对浏览器后台 tab 节流），但 PRD 的 useTimer 伪代码（第 90-103 行）仍使用 `setInterval` + `prev - 1` 固定递减。两套策略互斥。

**影响**: 如果按 PRD 伪代码实现，浏览器切到后台时计时会严重漂移，违背架构文档的精度承诺。

**修复**: PRD 和架构文档统一使用基于 `Date.now()` 的时间差策略。更新 PRD 伪代码以反映实际实现方式。

### 发现 3: 00:00 可见性矛盾 🟡

**问题**: 架构文档说"先显示 00:00 一帧，然后下一帧切换到新模式"。但 PRD 伪代码中 `if (prev <= 1)` 直接 switchMode + return nextModeDuration，用户永远看不到 00:00。

**修复**: 明确行为——当 timeLeft 降到 1 时，先返回 0（显示 00:00），在下一个 tick 再切换到新模式。这样用户能短暂看到 00:00，视觉上确认"时间到了"。

### 发现 4: progress 和 totalTime 在返回类型中缺失 🟡

**问题**: 架构文档组件树中 TimerDisplay 需要 `progress` prop，但 useTimer 返回类型中未声明 `progress` 和 `totalTime`。

**修复**: 与发现 1 一并修复，扁平返回结构中包含 `progress` 和 `totalTime`。

### 发现 5: 第 1 轮审查的"修复"未落实到 PRD 🔴

**问题**: 审查报告问题 2 声称"修复了计时器精度策略"，但 PRD (`04_番茄钟计时器.md`) 的 useTimer 伪代码从未更新，仍使用 `prev - 1` 固定递减。

**修复**: 同步更新 PRD 中的 useTimer 伪代码。

### 发现 6: 快速连续点击"开始"未处理 🔴

**问题**: 如果 isRunning 为 true 时再次点击"开始"，`start()` 直接 `setRunning(true)`，没有幂等防护。可能创建重复的计时逻辑。

**修复**: `start()` 增加守卫：`if (isRunning) return`。

### 发现 7: 自动切换后 isRunning 状态未明确 🔴

**问题**: 25:00 工作结束 → 自动切换到 05:00 休息。切换后计时器应该自动继续运行还是暂停等用户手动开始？PRD 说"自动切换"但未说明切换后状态。

**决策**: 自动切换后**保持运行状态**（isRunning 保持 true）。理由：
1. PRD 说"循环计时"，暗示不间断
2. 用户故事强调"提高效率"，减少手动操作
3. 如果用户需要暂停，他们可以手动点暂停

### 发现 8: 测试覆盖缺口 🟡

**问题**:
- 缺少 `ControlButtons.test.tsx`：按钮显示逻辑（运行中显示"暂停"，停止显示"开始"）无测试覆盖
- `useTimer.test.ts` 缺少：快速双击开始（幂等性）、自动切换后 isRunning 状态、useEffect cleanup
- `TimerDisplay.test.tsx` 缺少：00:00 显示、切换瞬间的过渡状态

**修复**: 补充测试计划。

### 发现 9: 页面卸载时 useEffect cleanup 🟡

**问题**: 未提及组件卸载时清理计时器。如果用户关闭页面时计时器正在运行，未清理的 interval/timeout 可能导致内存泄漏。

**修复**: useTimer 的 useEffect 返回 cleanup 函数，确保组件卸载时清理所有计时资源。

## 第 3 轮审查: Design Review

> 审查模式: 视觉设计审查 — 色彩、排版、动画、响应式、交互状态
> 审查范围: docs/ 目录全部设计文档 + PRD

### 发现 D1: 色彩方案不完整 🟡

**问题**: 只定义了进度环颜色(#ef4444/#3b82f6)和背景环(#e2e8f0)，缺少页面背景、文字色、按钮颜色方案。

**修复**: 补充完整色彩规范——深色主题（slate-900背景），进度环、按钮、文字完整色值表。

### 发现 D2: 进度环尺寸缺少响应式 🟡

**问题**: 进度环默认 280px 固定尺寸，移动端(320px宽)几乎占满全屏。

**修复**: 补充响应式策略：桌面端 280px，移动端 `min(240px, 65vw)`；倒计时字号随视口缩放。

### 发现 D3: 排版层级未定义 🟡

**问题**: 倒计时仅"大字居中"描述，无具体字号、字重、间距。

**修复**: 补充排版规范表：倒计时 72px/700、标签 20px/500、按钮 16px/600，各区块间距 32px。

### 发现 D4: 按钮交互状态缺失 🟡

**问题**: 按钮未定义 hover、active、focus-visible 状态。

**修复**: 补充按钮交互状态：hover 亮度降低10%、active 缩放97%、focus-visible 白色聚焦环、transition 150ms。

### 发现 D5: 进度环缺少过渡动画 🟡

**问题**: strokeDashoffset 每秒突变，视觉跳帧不流畅。

**修复**: 补充 `transition: stroke-dashoffset 0.5s linear` 实现进度平滑过渡。

### 发现 D6: 模式切换颜色突变 🟢

**问题**: 进度环颜色(红→蓝)瞬间切换，无过渡。

**修复**: 补充 `transition: stroke 0.3s ease` 实现颜色平滑切换。

### 发现 D7: 倒计时数字等宽问题 🟡

**问题**: 未指定 tabular-nums，数字宽度不同导致文本抖动。

**修复**: 排版规范中补充 `tabular-nums` 等宽数字声明。

### 发现 D8: CSS 方案文档不一致 🟡

**问题**: PRD 未声明 CSS 方案；目录结构 App.css 与 Tailwind CSS 关系不明确。

**修复**: PRD 补充技术栈声明；App.css 注释明确为"Tailwind 无法覆盖的全局样式"。

| D3 | 排版未定义 | 补充排版规范表（字号/字重/间距） |
